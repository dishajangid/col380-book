\section{GPU Model of Computation}
\label{sec:gpu-model}

% Key points:
%  - Programs run partly on Host (CPU) and partly on Device (GPU)
%  - Host orchestrates the computation:
%      sends computation tasks, data, and code to the GPU
%  - Host OS system calls are used to allocate memory on the GPU (GDRAM)
%      even cudaMalloc is issued from host code, not from inside a kernel
%  - Device executes tasks specified as kernels
%  - Parallel kernel execution is possible via non-blocking launches (Streams)
%      multiple kernels can overlap if SMs are not fully occupied
%  - GPC execution hierarchy (finest to coarsest):
%      Thread → Warp (32 threads, lockstep) → Block (TB) → TPC → GPC → GPU
%      A kernel is the unit of computation for a GPC

A CUDA program runs partly on the \ppterm{host} (CPU) and partly on the
\ppterm{device} (GPU). The host orchestrates the entire computation: it
allocates memory on both sides, transfers data to the GPU, launches work, and
retrieves results. The device executes the compute-intensive portions of the
program, which are expressed as special functions called \ppterm{kernels}.

\begin{ppkey}
The host sends both data \emph{and} code to the device. Even device-side
heap memory must be allocated through host-level system calls---the host OS
issues the necessary driver calls on the program's behalf.
\end{ppkey}

\subsection{Host and Device}
\label{ssec:host-device}

The host machine is a conventional CPU system. Within a CUDA program, the
programmer writes host code (ordinary C/C++ that runs on the CPU) and device
code (kernel functions that run on the GPU) in the same source file, saved
with the \texttt{.cu} extension. The NVIDIA compiler \texttt{nvcc} separates
the two: device code is compiled for the GPU, and host code is passed through
to the system C++ compiler (typically GCC).

\subsection{The GPU Execution Hierarchy}
\label{ssec:execution-hierarchy}

The logical execution hierarchy inside a GPU, from finest to coarsest, is:

\begin{itemize}
  \item \ppterm{Thread} --- the smallest unit of execution. Each thread runs
    an instance of the kernel with its own register state.
  \item \ppterm{Warp} --- 32 threads that execute in lockstep on the same
    functional units. The warp is the hardware scheduling unit.
  \item \ppterm{Thread Block (TB)} --- a programmer-visible group of threads
    mapped to a single Streaming Multiprocessor (SM). Threads within a block
    can communicate through fast on-chip shared memory.
  \item \ppterm{TPC (Texture Processing Cluster)} --- a grouping of SMs.
  \item \ppterm{GPC (Graphics Processing Cluster)} --- a grouping of TPCs.
    A kernel is the unit of computation assigned to a GPC.
  \item \ppterm{GPU} --- the full device, comprising all GPCs.
\end{itemize}

\begin{ppnote}
Tensor Cores (four per SM) are not assigned per-thread. The SM warp scheduler
identifies warps whose instructions require tensor arithmetic and routes them
to an available Tensor Core. A warp of 32 threads is the basic execution unit
for a Tensor Core operation.
\end{ppnote}

\subsection{Parallel Kernel Execution and Streams}
\label{ssec:streams}

Multiple kernels can execute concurrently on the same GPU provided there are
unallocated SMs. Concurrent launches are achieved through \ppterm{CUDA Streams}:
an ordered queue of GPU operations in which operations across different streams
may overlap. By default every kernel launch uses the null stream, which
serializes launches; explicit streams are required for true concurrency.

\section{Kernels: Definition, Structure, and the \texttt{\_\_global\_\_} Qualifier}
\label{sec:kernels}


\subsection{Definition of a Kernel}
A \textbf{kernel} is the fundamental unit of computation on a GPU. It is a function executed on the \emph{device} (GPU) and invoked from the \emph{host} (CPU). At the hardware level, a kernel represents the unit of work assigned to a \textbf{GPC (Graphics Processing Cluster)}, and is executed in parallel across many threads.

% \end{itemize}

\subsection{Defining a Kernel with \_\_global\_\_}

\begin{ppprogram}{Example CUDA Kernel}{prog:cuda-vector-add}
\begin{lstlisting}[language=C++]
__global__ void vector_add(const float *A,
                           const float *B,
                           float *C,
                           int N)
{
    int i = blockIdx.x * blockDim.x + threadIdx.x;

    if (i < N) {
        C[i] = A[i] + B[i];
    }
}
\end{lstlisting}
\end{ppprogram}

\begin{ppnote}
The \verb|__global__| qualifier has precise meaning:
\begin{itemize}
    \item It is a \textbf{compiler directive} interpreted by \texttt{nvcc}.
    \item It instructs the compiler to compile the function for execution on the device and make it callable from the host.
    \item Such functions are executed on the device but invoked from the host.
\end{itemize}
\end{ppnote}

\subsection{Kernel Syntax Rules}

\begin{ppnote}
CUDA kernels follow strict syntactic and semantic constraints:
\begin{enumerate}
    \item \textbf{Return Type:} Always \verb|void|, since many threads execute concurrently with no single return value.

    \item \textbf{Parameter Passing:}
    \begin{itemize}
        \item Passed by value (e.g., \verb|int N|)
        \item Passed as pointers (e.g., \verb|float *A|)
    \end{itemize}
    Pointer arguments must refer to \textbf{device memory}.

    \item \textbf{SPMD Execution Model:}
    \begin{itemize}
        \item All threads execute the same program.
        \item Each thread operates on different data using its index.
    \end{itemize}

\begin{lstlisting}[language=C++]
int i = blockIdx.x * blockDim.x + threadIdx.x;
\end{lstlisting}

    \item \textbf{Boundary Checks:}
\begin{lstlisting}[language=C++]
if (i < N) {
    C[i] = A[i] + B[i];
}
\end{lstlisting}
    Missing this leads to undefined memory behavior.
\end{enumerate}
\end{ppnote}

\subsection{Kernel Invocation}

\begin{ppprogram}{Kernel Launch Syntax}{prog:cuda-launch}
\begin{lstlisting}[language=C++]
int threads_per_block = 256;
int blocks = (N + threads_per_block - 1) / threads_per_block;

vector_add<<<blocks, threads_per_block>>>(d_A, d_B, d_C, N);
cudaDeviceSynchronize();
\end{lstlisting}
\end{ppprogram}

\begin{ppnote}
\begin{itemize}
    \item \verb|<<<blocks, threads_per_block>>>| specifies grid and block size.
    \item Kernel launches are \textbf{asynchronous} by default.
    \item \verb|cudaDeviceSynchronize()| enforces completion.
\end{itemize}
\end{ppnote}

\subsection{Source File Organization}

\begin{ppnote}
\begin{itemize}
    \item Kernel and host code coexist in the same \verb|.cu| file.
    \item The CUDA compiler \texttt{nvcc} separates:
    \begin{itemize}
        \item Host code $\rightarrow$ host compiler
        \item Device code $\rightarrow$ GPU binary (PTX/SASS)
    \end{itemize}
\end{itemize}
\end{ppnote}

\subsection{Kernel Compilation and Loading}

\begin{ppnote}
\begin{itemize}
    \item Kernel binaries are \textbf{not} sent at compile time.
    \item They are transferred at \textbf{first invocation} (lazy loading).
\end{itemize}
\end{ppnote}

\begin{ppnote}
\textbf{Implications:}
\begin{itemize}
    \item First launch incurs overhead (loading / JIT compilation).
    \item Subsequent launches are faster.
\end{itemize}
\end{ppnote}

\subsection{Execution Mapping to Hardware}

\begin{ppnote}
\begin{itemize}
    \item Kernel $\rightarrow$ GPCs
    \item Thread blocks $\rightarrow$ SMs
    \item Threads $\rightarrow$ executed within SMs
\end{itemize}

A thread block executes entirely on a single SM, enabling efficient shared memory usage.
\end{ppnote}

\subsection{Common Pitfalls}

\begin{pppitfall}
\begin{itemize}
    \item Missing \verb|__global__| $\rightarrow$ compiled for host
    \item Using host pointers in kernel $\rightarrow$ invalid access
    \item Missing bounds check $\rightarrow$ memory corruption
    \item Assuming synchronous execution $\rightarrow$ race conditions
\end{itemize}
\end{pppitfall}

\section{Thread Indexing and the Bounds Check}
\label{sec:thread-indexing}

% Key points:
%  - SPMD execution model: every thread runs the same kernel code
%      but operates on a different data element via its unique index
%  - threadIdx.x alone is NOT globally unique — it resets to 0 in each block
%  - Globally unique 1D thread index formula:
%      i = blockIdx.x * blockDim.x + threadIdx.x
%  - For 2D grids (e.g. matrix kernels):
%      row = blockIdx.y * blockDim.y + threadIdx.y
%      col = blockIdx.x * blockDim.x + threadIdx.x
%  - Why if (i < N) is required:
%      blocks = ceil(N / threadsPerBlock)  [ceiling, not floor]
%      ceiling division over-provisions: last block may have idle threads
%      without the guard, surplus threads perform silent out-of-bounds writes
%  - Example: N=34, threadsPerBlock=16 → 3 blocks, 48 threads total,
%      threads 34–47 (14 threads) have no work and must be suppressed


\subsection{The SPMD Execution Model}
\label{ssec:spmd}

CUDA follows a \ppterm{Single Program Multiple Data} (SPMD) model: every
thread executes identical kernel code, but each thread operates on a different
data element determined by its unique thread index. The programmer writes the
computation for a \emph{single} element; the hardware replicates it across all
launched threads.

\subsection{Computing the Globally Unique Thread Index}
\label{ssec:thread-index}

Three built-in read-only variables identify a thread's position:

\begin{itemize}
  \item \ppinline{threadIdx.x} --- index within the block (resets to 0 in
    every block; \emph{not} globally unique).
  \item \ppinline{blockIdx.x} --- index of the block within the grid.
  \item \ppinline{blockDim.x} --- number of threads per block (same for all
    blocks in a launch).
\end{itemize}

The standard globally unique index formula for a one-dimensional grid is:
\[
  i \;=\; \texttt{blockIdx.x} \times \texttt{blockDim.x}
          \;+\; \texttt{threadIdx.x}
\]

For a two-dimensional grid (used in matrix kernels such as
Program~\ref{prog:tensor-cuda-naive}):
\[
  \texttt{row} = \texttt{blockIdx.y} \times \texttt{blockDim.y}
               + \texttt{threadIdx.y}, \quad
  \texttt{col} = \texttt{blockIdx.x} \times \texttt{blockDim.x}
               + \texttt{threadIdx.x}
\]

\subsection{Why the Bounds Check \texttt{if (i < N)} Is Required}
\label{ssec:bounds-check}

The number of blocks must be an integer, so block count is computed using
\emph{ceiling} division:
\[
  \text{blocks}
  = \left\lceil \frac{N}{\text{threadsPerBlock}} \right\rceil
  = \frac{N + \text{threadsPerBlock} - 1}{\text{threadsPerBlock}}
\]

Ceiling division may create surplus threads in the last block whose index
$i \geq N$. Without the guard \ppinline{if (i < N)}, those threads perform
silent out-of-bounds memory writes---no segmentation fault is raised.

\begin{pppitfall}
Omitting \ppinline{if (i < N)} when using ceiling block-count arithmetic
causes silent out-of-bounds writes to device memory. The corruption produces
no runtime error, making it one of the most common and hardest-to-diagnose
CUDA bugs.
\end{pppitfall}

\begin{ppexample}
$N = 34$, \ppinline{threadsPerBlock = 16}.
$\lceil 34/16 \rceil = 3$ blocks $\Rightarrow$ 48 threads total.
Threads $34, 35, \ldots, 47$ (14 threads in the last block) have no
corresponding array element and must be suppressed by the bounds check.
\end{ppexample}

Program~\ref{prog:vec-add-bounded} shows the corrected, production-style
vector addition kernel that incorporates both multi-block indexing and the
bounds check.

\begin{ppprogram}{Vector addition kernel with bounds check}{prog:vec-add-bounded}
\begin{lstlisting}[language=C++]
__global__ void vector_add(const float *A, const float *B,
                           float *C, int N)
{
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < N)
        C[i] = A[i] + B[i];
}
\end{lstlisting}
\end{ppprogram}

%  - Three qualifiers; each controls WHERE a function runs and WHO calls it

%  __global__
%    - Executes on device, called from host (defines a kernel)
%    - Return type must be void
%    - Invocation crosses the PCIe bus (host → GPU)

%    - Call stack on the device mirrors CPU behavior

%    - Typical use: helper/math utility functions called many times inside a kernel
%
%  __host__
%    - Executes on CPU, called from CPU — standard C++ function
%    - Default qualifier: implicit when no qualifier is written
%    - Can be combined with __device__ to compile both CPU and GPU versions

\section{Variable Qualifiers and the Memory Hierarchy}
\label{sec:var-qualifiers}

% Key points:
%  - Programmer (not hardware) is responsible for placing data
%    in the right memory space — a key performance design decision
%
%  __device__  (GDRAM — Global Device RAM)
%    - Resides off-chip in GDRAM
%    - Scope: all threads, all blocks
%    - Lifetime: entire application
%    - Latency: high (~200–800 cycles uncached)
%    - Capacity: large (GBs)
%    - Dynamic allocation: cudaMalloc from host code
%
%  __constant__  (Constant Memory)
%    - Resides off-chip in a dedicated constant memory region (cached)
%    - Read-only from device; host writes via cudaMemcpyToSymbol
%    - Scope: all threads; Lifetime: entire application
%    - Latency: low for broadcast (all threads in a warp read same address)
%              serialized (high latency) if threads read different addresses
%    - Capacity: 64 KB on most architectures
%    - Best for: read-only constants, filter coefficients, lookup tables
%
%  __shared__  (Shared Memory / On-chip SRAM)
%    - Resides on-chip in the SM's SRAM
%    - On modern GPUs: dual-configurable with L1 cache (same physical SRAM,
%      software-configurable split)
%    - Scope: threads within the same block only
%    - Lifetime: block lifetime (deallocated when block finishes)
%    - Latency: very low (~1–4 cycles)
%    - Capacity: small (~48–164 KB per SM, shared among all resident blocks)
%    - Tradeoff: faster but scarce — programmer must decide what to stage here
%    - Basis for tiling optimizations (covered in a future lecture)

\section{Kernel Invocation, Block/Grid Sizing, and Synchronization}
\label{sec:invocation}

% Key points:
%
%  Chevron (execution configuration) syntax:
%    kernel_name<<<gridDim, blockDim>>>(args);
%    gridDim  = number of blocks  (dim3, can be 1D/2D/3D)
%    blockDim = threads per block (dim3, can be 1D/2D/3D)
%
%  Computing the number of blocks — ceiling formula:
%    blocks = (N + threads_per_block - 1) / threads_per_block
%    floor division would leave the last few elements uncovered (silent bug)
%    ceiling over-provisions a few idle threads in the last block (safe with guard)
%
%  Hardware limit on threads per block:
%    determined by SM register file size
%    e.g. NVIDIA A100: max 2048 threads/block
%    hardware designers expose this as a tunable limit because optimal
%    block size is workload-dependent
%
%  cudaMalloc — heap allocation on the device (called from host)
%  cudaMemcpy — data transfer with explicit direction flag:
%    cudaMemcpyHostToDevice  (H2D): CPU → GPU before kernel
%    cudaMemcpyDeviceToHost  (D2H): GPU → CPU after kernel
%
%  cudaDeviceSynchronize():
%    kernel launches are asynchronous — host returns immediately after <<<>>>
%    cudaDeviceSynchronize() blocks the host until all GPU work is done
%    must be called before D2H cudaMemcpy to avoid a race condition
%
%  Choosing block count — workload-dependent tradeoff:
%    compute-bound: fewer, larger blocks → fewer SMs active → less inter-SM sync
%    memory-bound:  more blocks → more in-flight warps → better latency hiding
%    no universal answer — benchmark on target hardware
%
%  Inter-SM memory sharing:
%    within a block: threads share __shared__ memory on the same SM
%    across blocks:  only L2 cache and GDRAM are accessible to all blocks
%    there is NO cross-SM shared memory
%    cross-block communication requires GDRAM + a second kernel launch
